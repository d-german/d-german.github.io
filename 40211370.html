Hyland Clinician Window : Inversion of Control (IoC)

Created by Damon German, last modified on Sep 04, 2022

Instead of you being in control of when a piece of code is executed, you hand the control to something else. The flow of control is now inverted. It also known this as the Hollywood principle, “Dont call us, we will call you.”

Inversion of Control moves secondary responsibilities from an object to other objects that are dedicated to the purpose, supporting the Single Responsibility Principle. (Martin 209)

Examples

LINQ

In LINQ, you supply the lambda expression, but LINQ controls when and how often it gets called.

Windows Forms

In Windows Forms, the framework declares events on controls. You can then bind a method to the event by using a delegate.

In this example, the user is in control when the ui_CalculateButton_Click method is executed by clicking the ui_CalculateButton.

Unity Game Framework

In Unity, a game object must inherit from MonoBehavior and must have certain defined methods that the framework calls while the game is running. Like Update() for example.

IoC Containers

In dependency management, an object should not take responsibility for instantiating dependencies itself. Instead, it should pass this responsibility to another “authoritative” mechanism, inverting the control. (Martin 2009)

IoC is most well known in the context of IoC containers. These systems allow you to link the interfaces of your application to their implementations and retrieve an instance of a class by resolving all its dependencies. (McLean Hall, Gary) I think this might be why people use the words inversion of control, dependency injection and dependency inversion interchangeably, although there is a distinction.
