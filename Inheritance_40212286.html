Hyland Clinician Window : Inheritance

Created by Damon German, last modified on Feb 01, 2021

A feature of object-oriented programming languages by which classes may be specialized from more general superclass. Attributes and Method definitions from Superclasses are automatically acquired by the subclasses.

Entire class hierarchies can be made this way. From a base class, one or more classes can be derived. From each of these classes, zero or more classes can be derived, and so on.

Inheritance is the strongest, and most rigid, of all the source code relationships; consequently, it should be used with great care. (2017 Martin 88)



  • Two main types of inheritance: Implementation and Interface.
    • Easy to confuse if the programming language doesn't make the distinction.
    • The term inheritance is used both when a class derives from an non abstract (concrete) class to make a specialized version of it and when an abstract class is used as a starting point for different implementations. Would it be better if we had two different terms for these concepts instead of using the same term? (Shalloway 137)
    • Implementation: Derived classes inherit data and behavior (implementation) of the base class.
      • C++ approximates this using private inheritance
      • Implementation inheritance can degenerate into “like a” (Coplien 206) relationship when the derived class violates Liskov’s substitution principle.
        • This is only a problem when trying to use the relationship polymorphically. (Coplien 237)
      • When inheritance is used as “is a”, software reuse is an incidental (but important!) benefit. However, reuse is often used to justify degeneration into “like-a” relationships. This may have a short-term reuse benefit, but it may not encapsulate change in the long term. (Coplien 251)
    • Summary:
      •  If your primary purpose of inheritance is reuse, then inheritance is being used, incidentally. This “incidental inheritance” could give you troubles if trying to use polymorphically. This is important because it is polymorphism that aids in your design to be adaptable, maintainable, extendable and testable.
      •  If your main purpose of inheritance is proper subtyping (Liskov substitution) then reuse is an incidental benefit.
    • Interface inheritance: Derived classes inherit only the public interface of the base class and must provide its own implementation.
      • Approximated in C++, by inheriting from pure abstract classes.
      • C# makes this clear by providing interfaces.
    • Of course, you can have a mixture of both.
    • (GOF 16) Refers to this as Class vs Interface Inheritance.
  • When a class is derived from a base class, I say that the derived class has an “is a” relationship with the base. However, when a class implements an interface, I say that the implementing class has a “can-do” relationship with the interface. (Jeffrey Richter)
  • If it is possible to use one of  A's descendants in all the places where class A  is used,  then  we say that the classes are behaviorally compatible. (Jacobson 64)
  • If the descendant is modified in such a way that it is no longer behaviorally compatible with its parent (that is, if the parent class can no longer be exchanged with the descendant), the class is said to be specialized. (Jacobson 64)
