Hyland Clinician Window : Imperative vs Functional (Declarative, Expressive) Style of Programming

Created by Damon German, last modified on Jan 03, 2023

Say we want to sum the elements of an int array named _values (the data source). You might use the following code:


This loop specifies precisely how to add each array element’s value to the sum—with a for iteration statement that processes each element one at a time, adding its value to the sum. This technique is known as external iteration—because you specify how to iterate, not the library—and requires you to access the elements sequentially from beginning to end in a single thread of execution. To perform the preceding task, you also create two variables (sum and counter) that are mutated repeatedly—their values change as the iteration is performed. (Detiel 868)

The problem with external iteration is that even in this simple loop, there are many opportunities for error. You could, for example, (Detiel 868)

  • initialize variable sum incorrectly,

  • initialize control variable counter incorrectly,

  • use the wrong loop-continuation condition,

  • increment control variable counter incorrectly or

  • incorrectly add each value in the array to the sum.

In functional (Declarative, Expressive) programming, you specify what you want to accomplish in a task, but not how to accomplish it. (Detiel 868)

Using LINQ

Using LINQ allows you to say, “Here’s a data source, give me the sum of its elements.” You do not need to specify how to iterate through the elements or declare and use any mutable (that is, modifiable) variables. This is known as internal iteration, because the library code (LINQ behind the scenes) iterates through all the elements to perform the task. (Detiel 868)

A key aspect of functional programming is immutability—not modifying the data source being processed or any other program state, such as counter-control variables in loops. By using internal iteration, you eliminate from your programs common errors that are caused by modifying data incorrectly. This makes it easier to write correct code. (Detiel 868)

Static Methods

The reason I always make a big deal out of being static is that when a method does not change instance fields, that’s a big deal. When a method is static, we don’t have to worry about it causing side effects and it increases thread safety. So this explicitly declares we’re not mucking with the instance fields. Also, that leaves the possibility of moving the method out to a better location and/ or making it into an extension method.
