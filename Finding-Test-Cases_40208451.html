Hyland Clinician Window : Finding Test Cases

Created by Damon German, last modified on Sep 04, 2022

The following is the process and tools I used to create test cases. It might seem excessive, but I assure you, I do this every time. This is because I lack confidence in my abilities without following a process. 

For simple methods, I sketch a truth table of the inputs, then analyze each row and write the expected result. I cross out any rows that are impossible or I don't care about. Finally, I create a test case for each relevant row on the table.

When things are more complicated beyond a couple of boolean values, I use a free mind mapping tool to help model the test cases and keep track of testing progress.





Equivalence Partitioning

Equivalence partitioning divides the input domain of a variable into groups, called equivalent classes, that represent interesting conditions. We expect two values of a variable belonging to the same equivalent class to produce the same test results (pass or fail).

An input variable has certain ranges of valid and invalid values. For example, an int variable for month ranges from 1 to 12 so any value less than 1 or greater than 12 is invalid.

For another example, say we want to compare two lists. There might only be 3 classes to consider: that All items in one list matches the items in the other list, Some items match or None of the items match. 

Equivalence partition can be subjective and application dependent. Given an input variable, different people may produce distinct sets of valid and invalid classes for different applications. (Dianxiang 350)

When generating Testing Considerations, you don't need to include every possible value that an input variable can take. Use equivalence partitioning to generate a manageable subset of interesting values and be sure to include values for the happy path and the error path.
