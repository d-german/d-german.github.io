Hyland Clinician Window : Two main types of inheritance: Implementation and Interface.

Created by Damon German, last modified on Feb 25, 2022

TBD Still working on this:


  1. Easy to confuse if the programming language doesn't make the distinction.
  2. We use the term inheritance both when a class derives from a non abstract (concrete) class to make a specialized version of it and when an abstract class is used as a starting point for different implementations. Would it be better if we had two different terms for these concepts instead of using the same term? (Shalloway 137)
  3. Implementation: Derived classes inherit data and behavior (implementation) of the base class.
    1. C++ approximates this using private inheritance (TBD)
    2. Implementation inheritance can degenerate into “like a” (Coplien 206) relationship when the derived class violates Liskov’s substitution principle.
      1. This is only a problem when trying to use the relationship polymorphically. (Coplien 237)
    3. When inheritance is used as “is a”, software reuse is an incidental (but important!) benefit. However, reuse is often used to justify degeneration into “like-a” relationships. This may have a short-term reuse benefit, but it may not encapsulate change in the long term. (Coplien 251)
  4. Summary:
    1.  If your primary purpose of inheritance is reuse, then inheritance is being used, incidentally. This “incidental inheritance” could give you troubles if trying to use polymorphically. This is important because it is polymorphism that aids in your design to be adaptable, maintainable, extendable and testable.
    2.  If your main purpose of inheritance is proper subtyping (Liskov substitution) then reuse is an incidental benefit.
  5. Interface inheritance: Derived classes inherit only the public interface of the base class and must provide its own implementation.
    1. Approximated in C++, by inheriting from pure abstract classes.
    2. C# makes this clear by providing interfaces.
  6. Of course, you can have a mixture of both.
  7. (GOF 16) Refers to this as Class vs Interface Inheritance.
