Hyland Clinician Window : Seam Interface

Created by Damon German, last modified on Jan 26, 2023

A Seam Interface is a design pattern in which an interface is created to isolate a complex dependency, to make it more testable and easier to replace or mock in test scenarios. It allows us to isolate the complex/hard-to-test dependencies and to control how they interact with the rest of the system.

The term "seam" comes from Michael Feathers' book "Working Effectively with Legacy Code", where he introduces the concept of "seamful design" as a way to design code that is more testable by creating seams, or small, focused interfaces, which allow you to test parts of the system in isolation.

public interface IFullTextDocumentDataProvider
{
	PageData GetPage(ISession session, Rendition rendition, int pageNumber, GetPageProperties properties);
}
internal class FullTextDocumentDataProvider : IFullTextDocumentDataProvider
{
	// Test wrapper for concrete complex dependency that should already be tested
	[ExcludeFromCodeCoverage]
	public PageData GetPage(ISession session, Rendition rendition, int pageNumber, GetPageProperties properties)
	{
		return new DocumentDataProvider(session as Session).GetPage(rendition, pageNumber, properties);
	}
}

The DocumentDataProvider class is a dependency that is difficult or impossible to mock in a test environment. Wrapping it in an interface, IFullTextDocumentDataProvider and creating an implementation of that interface, FullTextDocumentDataProvider, allows for the GetPage() method to be tested in isolation by providing a mock implementation of the interface. By wrapping the complex dependency in an interface, it allows to test the class depend on the DocumentDataProvider without having to use the real implementation of the DocumentDataProvider class and its dependencies.
