Hyland Clinician Window : Abstract Classes vs Interfaces

Created by Damon German, last modified on Aug 20, 2022

Compare and contrast abstract classes and interfaces. Why would you use an abstract class? Why would you use an interface?

One way to see the difference between the two is to compare programming languages that have explicit interfaces such as Java and C# and those that do not such as C++. C++ only has classes. In C++, you can simulate interfaces by using pure abstract classes. (GOF 17) Pure abstract classes in C++ are abstract classes that have only pure virtual functions. A pure virtual function is a function that has no implementation or body, it is pure signature. A virtual function is "made pure" by the "pseudo initializer" =0 (Stroustrup 598):


Note that there is nothing keeping a developer from updating Shape to contain a non-pure virtual function, but in that case it would no longer be an "Interface".

How would we write a "pure abstract class" in C#? In C# we declare a pure virtual function or method using the abstract keyword modifier. If we declare any of a classes' method abstract, then we must also declare the containing class as abstract. So the equivalent Pure Abstract class looks like this:

Like before, there is nothing stopping this class from being updated to contain virtual methods, i.e. methods that have implementation. 

Compare and contrast abstract classes and interfaces.

Since both are missing at least some implementation, neither can be instantiated (or newed up). This also means they do not need a public and/ or default constructor. Abstract classes may contain some instance fields and default implementation meant to be shared or overridden by derived classes. If an abstract class has instance fields, you can use a protected constructor to initialize them. C# allows you to declare an abstract base class constructor as public, but I do not recommend this. (FDG 95-96).

You can say that you inherit from a base class but implement an interface. Inheritance is an “is-a” relationship. We say “a cat is a mammal” and “a mammal is an animal.” In both cases it's common to call this an "is-a" relationship.  Some refer to implementing an interface as a "can-do" relationship.

C# structs can implement interfaces but cannot inherit from classes.

C# and Java does not allow derived classes to inherit from more than one base class (multiple inheritance), but it allows a class to inherit from a single base class and/or implement many interfaces.

Relationship to General Design Principles

"Program to an interface; not an implementation" (GOF 18), the word interface refers to both pure abstract classes or interfaces. We can restate this as "Program to an interface or (pure?) abstract class; not an implementation”.

"Prefer composition ("has-a") over class inheritance" (GOF 20). Here "class inheritance" is being distinguished from "interface inheritance" (GOF 17). "Interface inheritance" is equivalent to deriving from a pure abstract class or implementing an interface.

Relationship to SOLID Design Principles

Both pure abstract classes and interfaces enable us to follow the Open Closed Principle (OPC), and the Dependency Inversion Principle (DIP). 

Abstract classes can contain implementation. It is possible that derived classes may override the default behavior in a way that makes them not fully substitutional for the base class. When this happens, they lose their ability to be used polymorphically. The “is-a” relationship had degenerated into a “is-like-a” relationship (Coplien 251). Here, it violates Liskov Substitution Principle (LSP). Since interfaces lack implementation, in theory its harder to violate the LSP using interfaces.

Since we typically use interfaces for unrelated classes, they tend to have a leaner set of members. Remember, each member of an interface must be implemented in derived classes, but we should not force clients to implement methods they do not use. This “interface pollution” or “fat interface” tends to violate the Interface Segregation Principle (ISP). 

Example Fat Interface


A class that implements this interface must implement every method. The problem is that the implementing class would most likely violate the SRP. The class A implementations would need to be updated for different reasons and for different times than the B or C implementations.


Using Interfaces with base classes

Sometimes it’s good to use an Interface along with a base class so developers can choose which is best for them. They can get free default implementation if they use the base class. This is a common practice in the .Net framework as we can see in System.Collections Namespace.

List ADT example of AC or I

An ADT like a Stack is an example how a classes public interface - the public constructors and methods belong together (high cohesion) and represent the essential properties of an object. (Dianxiang Xu 86)

Stack(), Push(), Pop(), Peek(), IsEmpty()

With a Stack, I would prefer to implement it as an abstract class. It would be OK if we forced clients to implement every member since these methods belong together and their implementation should not violate the SRP.

Summery

If you public interface contains more than a few members (Don't know how many) and they belong together (high cohesion) then represent them as an abstract class. If the members do not relate well (low cohesion) then we should divide them into multiple smaller Interfaces.

**Beginning with C# 8.0 on .NET Core 3.0, you can define an implementation when you declare a member of an interface. The most common scenario is to safely add members to an interface already released and used by innumerable clients. This includes static fields and methods and excludes instance fields. You can provide a body to interface methods, but the method does not have access to any instance fields.
